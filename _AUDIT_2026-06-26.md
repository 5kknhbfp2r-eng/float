## [CRITICAL] SPAC/IPO carried-float replay has NO staleness signal for the 8-K events that actually change it (de-SPAC, redemptions, IPO lock-up/dilution)
- **module/where:** architecture — `engine/recipe_cache.py:29-37,124,154-155`  (accuracy)
- **verdict:** confirmed (finder confidence high)
- **what:** For basis in ('spac','ipo') replay() returns the carried float_at verbatim (line 154-155). Its only staleness gate is is_stale(), which scans STRUCT_FORMS (13D/13G/proxies) + OS_EVENT_FORMS (424B) only (lines 29-37). The events that change a SPAC's redeemable public float (the de-SPAC business-combination close, and the running redemption count as the deadline approaches) and an IPO's effective free float (lock-up expiry, follow-on dilution, founder share conversions) are reported in 8-K, which is NOT in either form set (confirmed: '8-K' in STRUCT_FORMS+OS_EVENT_FORMS == False). _split_in_window (the only 8-K reader) is unreachable for these bases because the spac/ipo branch returns BEFORE line 168.
- **impact:** After a SPAC completes its business combination, the redeemable-public-share count collapses (most public holders redeem) while replay keeps returning the pre-combination float_at as status 'ok' (FREE). This emits a confidently-WRONG float on the deterministic path with no defer — exactly invariant (1)'s critical class. Magnitude can be 100%+ (a 100M-redeemable-share SPAC that redeems down to 5M still reports ~100M). No spac recipes are in the current 33-recipe cache, but save_recipe/basis_route fully support basis='spac' and the planned full-236 warm includes SPAC-archetype tickers, so the path will be exercised.
- **fix:** For basis in ('spac','ipo'), either (a) treat ANY 8-K filed in (derived_day, day] as stale (defer to LLM), or (b) re-derive the redeemable/offering count from the latest periodic cover each day (basis_route already parses 'subject to possible redemption' from the periodic) and band-check it against float_at, deferring on any change. Do not return a fixed float_at for these bases without an 8-K-aware staleness gate.
- **verifier note:** Two corrections, both AGGRAVATING the finding rather than weakening it. (1) The finding claims 'No spac recipes are in the current 33-recipe cache' and frames exposure as only 'planned'. WRONG/understated: there is a LIVE basis='ipo' recipe RIGHT NOW — BMGL (cik 2004489, derived 2025-05-14, float_at=8.5818, os_at=18.7858), recipe_cache.RECIPES counts {'normal':32,'ipo':1}. The vulnerable branch is exercised today, not hypothetically. (2) The finding omits the partial backstop and the suggested fix is INCOMPLETE. The O/S anchor band (lines 139-153) DOES run before line 154, so replay defers when re-fetched O/S drifts >±30% of os_at — this catches large total-O/S jumps (e.g. a big de-SPAC issuance) but NOT the redemption/lock-up/founder-conversion changes that move the public-float COMPOSITION while keeping total O/S within ±30%. So the hole is real but realized only on the in-band subset (still critical-class). The suggested fix (a) 'treat any 8-K in (derived_day,day] as stale' must ALSO include 6-K: the only live ipo recipe (BMGL) is a foreign private issuer that files 6-K, not 8-K (verified: 16 6-K filings, 0 8-K, after derivation), and `_split_in_window` likewise only reads 8-K. A fix that gates on 8-K alone leaves the single recipe currently in the cache fully unprotected. Correct fix: for basis in ('spac','ipo'), defer on ANY 8-K OR 6-K filed in (derived_day, day].

## [CRITICAL] resolve_cik_edgar picks the highest-relevance EFTS hit, not the point-in-time filer — returns the WRONG entity for reused tickers
- **module/where:** gather_edgar — `engine/edgar.py:117-140`  (accuracy)
- **verdict:** confirmed (finder confidence high)
- **what:** resolve_cik_edgar bounds the EFTS query by enddt=asof (date axis is safe) but then returns the FIRST display_names match in the hit list. EFTS sorts hits by relevance _score, NOT by date. For a ticker that two CIKs have used (ticker reuse), the newer/more-active entity frequently scores highest while still having filings inside the enddt window, so the resolver returns it instead of the entity that actually used the ticker at as_of. Verified live: edgar.resolve_cik_edgar('MTAL','2026-06-04') returns 2107724 (Metals Acquisition Corp. II, top _score 23.0, file_date 2026-04-08) instead of the point-in-time MTAL holder 1950246 (Metals Acquisition Ltd). The top EFTS hit for that query is the wrong, newer entity. Only the hardcoded CIK_OVERRIDES={'MTAL':1950246} masks this one case; every other un-overridden reused ticker resolves to whichever CIK EDGAR scores highest, biased toward the most recent active filer.
- **impact:** resolve_cik() (float_gather.py:182) routes to this fallback whenever (a) the static map misses, or (b) the point-in-time guard at float_gather.py:202 fires because the mapped CIK has no filing <= asof. The returned wrong CIK then drives _assemble/query/_pick_own/_current_os for that (ticker,asof): the entire dossier (O/S, ownership table, control holders) is built from a DIFFERENT company's filings, producing a confidently-wrong float for the labeled ticker-day, with no defer signal. This is a wrong-entity accuracy bug affecting the float for any reused ticker not in the 1-entry override map.
- **fix:** Do not trust _score ordering. Among hits whose display_names match (TICKER), select the CIK whose filing file_date is the MOST RECENT but <= asof (the entity actually using the ticker at as_of), or collect all candidate CIKs and pick the one with a filing closest-to-and-before asof. At minimum, when multiple distinct CIKs match, prefer the one with the latest file_date <= asof and flag/defer when candidates are ambiguous rather than silently taking the first.
- **verifier note:** The mechanism and fix are correct, with one scoping nuance the finding understates only slightly: the bug bites only when the wrong (higher-_score) CIK ALSO has at least one filing inside the [startdt=2022-01-01, enddt=asof] window. The enddt cap (line 122) is a real partial safety — for an asof before the newer entity's first filing, the wrong CIK is filtered out by the date axis (verified: resolve_cik_edgar('MTAL','2024-06-15') correctly returns 1950246, because 2107724's earliest filing is 2026-04-08, outside that window). So the wrong-entity result occurs specifically when BOTH entities have filings <= asof and the unwanted one scores higher — common for any reused ticker queried at an asof after the new entity began filing, but not literally every reused ticker/date. The suggested fix is sound: among hits whose display_names match (TICKER), pick the CIK with the latest file_date <= asof (the entity actually using the ticker at as_of), and when multiple distinct CIKs match within the window, defer/flag rather than silently taking the first _score-ranked hit. Note also the enddt query bound caps filing DATE but the function then still trusts _score ordering — the fix must explicitly re-rank/collect by date, since EFTS will not do it.

## [CRITICAL] 13F-filer name match accepts a single shared token, auto-classifying a control person/entity as passive (false exclusion)
- **module/where:** holder_registry — `engine/holder_registry.py:91-105, 119-124`  (accuracy)
- **verdict:** confirmed (finder confidence high)
- **what:** lookup_13f resolves a holder to a CIK using the predicate `set(norm(dn).split()) >= ktok or (set(norm(dn).split()) & ktok and len(ktok) >= 2)` (line 91). The second branch matches on ANY single shared distinctive token whenever the holder name has >=2 tokens. So a holder name sharing one token with a registered 13F manager resolves to that manager's CIK; the 13F-HR presence on that wrong CIK (line 102) then marks the holder passive and caches it (lines 122-124). Verified live: lookup_13f('George Soros') returns (True, 1029160) — CIK 1029160 is Soros Fund Management, a 13F-HR filer — and classify('George Soros','13G') returns ('passive','13F-filer') and writes that verdict to the registry. The shared token 'soros' alone drove the match.
- **impact:** A control person/entity (e.g. a named individual or family vehicle) on a 13G is auto-classified passive and excluded from the control block computation. A false-passive => the holder is KEPT in free float when it should be excluded as a control affiliate; conversely if the recipe treats classified-passive holders as kept and control as excluded, the misclassification corrupts which holders feed dno_M/control_M, producing a wrong (free, 'ok') float with no LLM deferral. This is the exact false-passive -> wrong-float accuracy invariant violation. CIK 1029160 also files SC 13D, proving a 13F-HR filer is not inherently passive on every stake.
- **fix:** Require the resolved FILER name to match the holder strongly before trusting its 13F status: drop the single-shared-token branch (require token-superset, or require the holder's full normalized token set to be a subset of the filer's, and reject person-name holders / family/trust/holdco tokens). Additionally, only treat 13F-HR presence as passive when the holder's own resolved CIK is the 13F filer AND the current filing is a 13G — never infer passivity for a CIK that also files 13D for the same issuer.
- **verifier note:** The finding and mechanism are correct; the suggested fix is directionally right but incomplete in two ways. (1) There is a COMPOUNDING defect the fix must also close: in classify(), the REGISTRY lookup (lines 113-114) runs BEFORE the 13D guard (line 115). I verified that once 'george soros' is cached passive via the 13G path, classify('George Soros','13D') returns ('passive','registry') — the 13D-active safeguard is bypassed for every subsequent filing of that name. So even dropping the single-token branch is insufficient if a poisoned entry already exists or is written; the 13F-filer auto-passive must never be cached for a name whose resolved-filer match is weak, OR the 13D guard must precede the registry hit for entries sourced from '13F-filer'. (2) The core match defect is that norm() collapses 'Soros Fund Management LLC' to a single token 'soros', so even a token-superset requirement (holder tokens subset of filer tokens) STILL matches 'soros' >= 'soros'+'george'? no — superset would require {soros} >= {george,soros} which is False, so requiring filer-tokens superset-of holder-tokens, i.e. ktok subset of dn-tokens, would correctly REJECT 'George Soros' (since 'george' is absent from the filer name). That is the right predicate: require ktok <= set(norm(dn).split()) (holder's full token set is a subset of the filer name), and additionally reject person-name / family / trust / holdco holders before trusting 13F status. Confirmed REAL and CRITICAL.

## [HIGH] Reverse-split detector misses 'consolidation' phrasing (foreign reverse splits) and requires a ratio token, re-opening the HUSA pre-split-O/S failure for foreign issuers
- **module/where:** architecture — `engine/recipe_cache.py:24-25,111`  (accuracy)
- **verdict:** confirmed (finder confidence high)
- **what:** _split_in_window gates on _RS (matches only 'reverse [stock] split') AND _RS_RATIO both being present (line 111). It omits 'consolidation' / 'subdivision' — the term UK/foreign issuers use for a reverse split. By contrast float_gather.SPLIT_RE (line 89-92) deliberately includes 'consolidation|subdivision'. Confirmed: _RS.search('share consolidation') is False and _RS.search('1:8 reverse split on a basis') works only because the ratio happens to co-occur; an 8-K that announces 'effected a reverse stock split' with the ratio in a separate non-adjacent context still relies on _RS_RATIO hitting somewhere in the text.
- **impact:** A foreign issuer that does a 1-for-N share consolidation files an 8-K (or 6-K) phrased as 'consolidation' — undetected by _split_in_window, so replay proceeds. XBRL dei:EntityCommonStockSharesOutstanding keeps reporting the PRE-consolidation count until the next periodic (the exact stale-XBRL mechanism the guard exists to catch), and the anchor band passes because os_at was also pre-split. Result: float computed on a share base N times too large -> a confidently-WRONG free float (the HUSA 899% mode the STATUS claims is closed, but only for the literal phrase 'reverse split'). - **fix:** Reuse float_gather.SPLIT_RE in _split_in_window instead of the local _RS/_RS_RATIO pair (it already covers reverse/forward/consolidation/subdivision), and add '6-K'/'6-K/A' to the form list so foreign consolidations are scanned. - **verifier note:** The suggested fix is correct to add 6-K/6-K/A to the form list and to add consolidation/subdivision vocabulary, but it is WRONG that reusing float_gather.SPLIT_RE "drops the requirement that a separate ratio token also match." SPLIT_RE itself still requires a ratio token immediately adjacent to the split/consolidation word: SPLIT_RE.search('effected a reverse stock split')=False and SPLIT_RE.search('every 8 ordinary shares were consolidated into 1')=False. So swapping in SPLIT_RE only fixes the consolidation-vocabulary miss, not the no-adjacent-ratio miss. To truly close the 'effected a reverse stock split/consolidation' case the gate must match the keyword alone, e.g. re.compile(r"reverse\s+(?:stock\s+)?split|\b(?:share\s+)?consolidation\b|\bsubdivision\b", re.I) with NO required ratio co-occurrence (accuracy-safe: at worst it over-defers a benign 8-K/6-K to the LLM, a cost not an accuracy hit). Also note severity is borderline high vs medium: the mechanism is accuracy-critical (wrong free float marked 'ok'), but realized blast radius is narrow (requires a foreign issuer doing a consolidation announced via 6-K or via 'consolidation'-worded 8-K inside an active recipe window; current cache has only ~1 ADS issuer). I keep it at high because under-deferral producing a confidently-wrong float is the exact class the audit weights >> cost.

## [HIGH] Registry lookup precedes the 13D-active gate, so a 13F-auto-cached entity that later files a 13D is returned passive (control block kept in float)
- **module/where:** architecture — `engine/holder_registry.py:113-124`  (accuracy)
- **verdict:** confirmed (finder confidence med)
- **what:** classify() checks REGISTRY first (line 113) and returns the cached class BEFORE the 13D form-type gate (line 115). The 13F-filer path auto-writes the entity as class='passive' (line 122) keyed on norm(name) with no form context. So once a registered 13F manager is auto-cached passive via a 13G context, a SUBSEQUENT 13D filing by that SAME manager on any stock returns ('passive','registry') and never reaches the 13D-active gate. This is precisely the MTR/CFSB/RILYT lesson the gate at line 116 was written to enforce, but the auto-cache short-circuits it.
- **impact:** A 13F institution that goes activist and files a 13D (a real control stake) is classified passive -> KEPT in the float instead of excluded -> float too HIGH on the deterministic exclusion path. - **fix:** Move the 13D-active gate ABOVE the registry lookup for auto-derived passive entries: if form_type contains 13D AND the registry entry's source is in ('13F-filer','keeplist'), return ('judge','13D-active') rather than trusting the auto-passive cache. - **verifier note:** The suggested fix is correct and complete: move/duplicate the 13D-active gate ABOVE the registry lookup but make it conditional on the entry's source being a heuristic one — i.e. for a registry hit whose source is in ('13F-filer','keeplist'), a 13D/13D-A form_type must override to ('judge','13D-active'); a deliberate set_class verdict (source 'llm'/'llm-recipe'/'manual') should still win since that is a reviewed judgment. ## [HIGH] Solver double-counts the SAME holder (multiple source/amendment rows summed) -> spurious exclusion in the recovered formula
- **module/where:** formula_probe — `engine/_formula_probe.py:510-532`  (accuracy)
- **verdict:** confirmed (finder confidence high)
- **what:** solve() builds itertools.combinations over the full candidate list and SUMS the selected rows' shares. Candidates for one holder routinely appear as several rows with DIFFERENT share values: thirteen_dg_candidates() deliberately emits each distinct 13D/13G amendment block per holder (line 329-359), form4_candidates() adds a Form-4 holding, prospectus_candidates()/major_holders() add a table row, and holders() adds a `~basic` ex-warrant variant (line 84-87). Nothing dedups candidates by holder IDENTITY -- the only dedup anywhere in the solve paths is by share VALUE (round(c[1],2): line 600-604 in solve_from_sources, and the ex0/ex value-sets at lines 688/725/735 in _recover). So solve() is free to pick two or more rows of the same holder and add them. Empirically, on the existing _formula_db.json (150 tickers) 33 of the 150 successful (clean/weak/near) recoveries reach their fit by summing one holder twice or more, totaling ~232M of spurious excluded shares. Examples: PITA status=clean sums Smilodon Capital 9.744M(17.6%) + 5.273M(30.2%) = 15.0M for one entity (the two percents are mutually exclusive -> at most one block is the real current holding); MRLN status=weak err=0.0001 hits its 'exact' fit only by double-counting three holders (George Matthew Spencer 15.422+14.891, Quiet Capital 13.84+12.896, Bleichroeder 8.801+8.758, ~36.5M spurious); AEI sums Chan Heng Fai five separate 13D/13G amendment blocks (~26.7M spurious).
- **impact:** Each extra same-holder row added to the chosen combo overstates the recovered exclusion by that row's share count, i.e. excl_recovered = group + sum(rows) is too LARGE by the duplicated block. Because excl=os-float, an overstated exclusion = an understated float. The over-fit is reported as status clean/weak with a near-zero err, so a coincidental double-count masquerades as a recovered formula. _formula_db.json is the Phase-1 design artifact (COST_OPTIMIZATION.md cites it to size the induced-rule / per-ticker recipe); a formula whose exclusion is inflated by a duplicated holder, if seeded into a recipe's dno_M/control_M, yields a confidently-too-LOW float on every replay day. The thirteen_dg_candidates docstring (line 330-333) states the intent is for the solver to PICK an older block among amendments, but at-most-one-per-holder is never enforced.
- **fix:** Tag every candidate row with a holder-identity key (e.g. frozenset(wp._caps(name)) as already used by det_float strat_13dg line 211, stripping the `~basic`/`(Filed by)` suffixes) and forbid solve() from selecting two rows sharing that key in one combo -- either prune to the single best row per holder before combinations, or skip any combo whose selected rows have a duplicate identity key. This preserves the 'pick the older/ex-warrant block' choice (still one row per holder) while eliminating the summation.
- **verifier note:** The mechanism, the 33/150 count, and all three named examples are independently confirmed, so the finding stands. Two refinements. (1) Scope/severity nuance: this defect currently lives in the formula-PROBE / _formula_db.json (the Phase-1 design artifact), which exists and reproduces. The path to a live wrong replay float requires the over-fit formula to be promoted into a recipe's dno_M/control_M -- a downstream step not exercised in this module. So it is a latent accuracy defect (a wrong-too-low exclusion masquerading as a near-zero-err 'clean'/'weak' fit), not one already emitting wrong floats through replay today; high is defensible (>20% of successful recoveries affected, err-masking hides it), bordering medium only because impact is gated on recipe seeding. (2) The suggested fix is correct in direction but incomplete: (a) the identity key must strip BOTH the '~basic' suffix (line 87 uses nm[:18]+'~basic') AND any '(Filed by)'/'(F'-style truncated suffix, and must be applied in the _recover path's candidate list too -- most empirical dups (MRLN, AEI are group='exo') come through _recover at lines 676-748, NOT only solve_from_sources; (b) prefer SKIPPING any combo whose selected rows share an identity key over 'prune to single best row before combinations' -- pruning up-front can discard the legitimately-chosen OLDER/ex-warrant block (the OBAS/Capri 'pick the older 13D' intent at line 331), whereas a per-combo skip preserves the solver's freedom to choose which single block while forbidding the sum.
